## Implementing `Collector: Allocator`: why arena3 was needed and what problems came up (2/2)

Note author: shruti2522

This continues from the previous note. Writing `unsafe impl Allocator for MarkSweepGarbageCollector` ran into several problems, some in the arena design itself and some from Rust rules around ownership, reentrancy, and lifetimes. Below is each problem and what fixed it.

### Why arena2 could not be used

arena2 is a `LinkedList` of fixed size pages. Every slot has an 8 byte header that stores allocation chain pointers, and the collector checks each slot's header to decide if it is alive. There is no distinction between typed GC objects and raw byte slabs on the same page.

`Allocator::allocate` gives back raw bytes. When a `Vec<T, &MarkSweepGarbageCollector>` asks for its backing buffer, that buffer has no `GcBox` header. Under arena2, a `GcBox<Node>` and a Vec buffer could end up on the same page mixed together. The sweep phase would then walk into the Vec buffer looking for a `GcBox` at each slot and produce corrupt results. There was also no per page count of raw allocations, so `drop_dead_arenas` would either never free a page that only held raw slabs, or free it too early while the Vec was still in use

arena3 fixes this by keeping two separate page lists: `typed_arenas` for GC objects and `raw_arenas` for raw byte allocations. The mark and sweep phases only ever look at `typed_arenas` and never touch raw pages. Each raw page has an `active_raw_allocs: Cell<usize>` counter. `try_alloc_bytes` adds to it and `dealloc_bytes` subtracts from it, and `drop_dead_arenas` frees the page once it reaches zero. On the typed side, a per-page bitmap (one bit per slot) tracks liveness without storing anything inside each slot. This keeps all slots in a page the same size, which is needed for the size-class `alloc_cache` index to work correctly and for `grow` to extend a Vec buffer in place when the new layout stays in the same size class

### Problems that came up during the implementation

**`&self` vs `&mut self`.** `Allocator::allocate` takes `&self`, but all arena code used `&mut self`. Using `Arc`/`Mutex` to work around this would break `no_std`

Fix: the arena field was changed to `RefCell<ArenaAllocator<'static>>`. `allocate()` calls `borrow_mut()` for the duration of the allocation and drops the borrow right after. Because the collector is single threaded there is never a second caller trying to borrow at the same time

**Reentrancy.** If `allocate()` started a collection to free up space, `collect()` would try to call `borrow_mut()` on the same `RefCell` while `allocate()` still held it, causing a runtime panic. 

Fix: the threshold check and the `collect()` call happen before the allocator borrow is taken:

```rust
if self.collect_needed.get() && !self.is_collecting.get() {
    self.collect_needed.set(false);
    self.collect(); // borrows and releases the arena internally
}
let result = self.allocator.borrow_mut().try_alloc_bytes(layout)?;
```

A debug panic also fires if `allocate()` is called while `is_collecting` is set, catching any future code that reenters through an unexpected path.

**`deallocate` leak risk.** Bump allocators cannot free individual slots. Fix: arena3's `active_raw_allocs` counter gives `drop_dead_arenas` a way to reclaim an entire raw page once every allocation on it has been released. This is not per slot free, but it is enough for the bump-allocator model.

**Trace integration.** A plain `Vec<T, &MarkSweepGarbageCollector>` is invisible to the mark phase. If it holds `Gc<T>` values inside a GC managed object, those values look unreachable and get swept

Fix: `GcAllocVec<T: Trace>` and `GcAllocBox<T: Trace>` wrap `allocator_api2::Vec`/`Box` and implement `Trace` by visiting each element. Any heap-resident collection that holds `Gc<T>` values must use one of these wrappers. Plain data rooted at the stack or a `Root` boundary needs no wrapper.

**Hidden `'static` in `alloc_gc_node`.** `Collector::alloc_gc_node` used to return `ArenaPointer<'static, GcBox<T>>`, which silently extended the lifetime inside the implementation with no visible `unsafe` at the call site

Fix: the return type is now `ArenaPointer<'gc, _>` tied to `&'gc self`, making the lifetime explicit. Call sites that need `'static` storage, `Root::new_in`, `WeakGc::new_in`, and `WeakMap::insert`, now call `unsafe { ptr.extend_lifetime() }` directly with a safety comment, yhe unsafe operation is visible exactly where it happens.
